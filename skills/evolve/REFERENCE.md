# Evolve — Reference

## Interview Dimensions

### 1. The change
- What exactly is being added or changed? One precise sentence.
- Why now — what triggered this? A user complaint, a gap noticed, a competitor move?
- Is this a net-new capability, a modification of existing behavior, or a nuance on top of something that already exists?

### 2. Audience fit
- Does this serve the existing audience, or does it attract a new segment?
- If it attracts new users, does that create tension with current users or the product's positioning?
- Is the existing audience asking for this, or is this the builder's hypothesis about what they need?

### 3. Product fit
- Does this reinforce what makes the product distinctive, or does it add generic functionality competitors already have?
- Does it expand the product's scope in a way that makes it harder to explain?
- If removed, would the product be meaningfully weaker — or just slightly smaller?

### 4. Scope
- What is the minimum version of this change that delivers real value?
- What is explicitly out of scope — what related things won't be touched?
- Are there edge cases or user segments this intentionally won't serve?

### 5. Competitive angle
- Do competitors have something like this? If yes, what's different about this approach?
- Is the goal to match a competitor, surpass them, or take a different angle entirely?
- Does this close a gap users complain about, or open a gap competitors don't serve?

### 6. UX impact
- Where in the existing product does this surface? A new screen, an addition to an existing one, a background behavior?
- Does this change any existing workflows users already know?
- What does the core interaction look like — what does the user do, and what happens?

### 7. Roadmap priority
- Is this a must-have (product is incomplete without it), should-have (clear value, not urgent), or nice-to-have?
- Does this block or depend on anything else in the roadmap?
- If the team could only ship one more thing, would this be it? Why or why not?
